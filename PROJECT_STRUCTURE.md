# مستندات ساختار پروژه (Project Structure Documentation)

این فایل راهنمای ساختار پوشه‌بندی و معماری استفاده شده در این پروژه است. این ساختار بر اساس اصول **Clean Architecture** و **Domain-Driven Design (DDD)** پیاده‌سازی شده است تا قابلیت نگهداری، تست‌پذیری و توسعه‌پذیری را افزایش دهد. شما می‌توانید از این الگو در پروژه‌های NestJS دیگر استفاده کنید.

## نمای کلی (Overview)

```
.
├── prisma/                  # تنظیمات و اسکیماهای دیتابیس (ORM)
├── scripts/                 # اسکریپت‌های کاربردی و نگهداری (مثل تست بار، مانیتورینگ)
├── src/                     # کدهای اصلی برنامه
│   ├── modules/             # ماژول‌های مستقل (Feature Modules)
│   ├── app.module.ts        # ماژول ریشه
│   └── main.ts              # نقطه ورود برنامه
├── test/                    # تست‌های E2E
├── Dockerfile               # تنظیمات داکر
└── package.json             # وابستگی‌ها و اسکریپت‌های اجرا
```

## معماری ماژول‌ها (Module Architecture)

هر ماژول در پوشه `src/modules` (مانند `activity`, `user` و ...) به چهار لایه اصلی تقسیم شده است. این تفکیک باعث می‌شود منطق بیزنس از جزئیات فنی جدا شود.

### ساختار داخلی یک ماژول

```
src/modules/feature-name/
├── application/             # لایه کاربرد (Use Cases)
│   ├── commands/            # عملیات‌هایی که وضعیت سیستم را تغییر می‌دهند (CQS - Write)
│   ├── queries/             # عملیات‌های خواندن اطلاعات (CQS - Read)
│   ├── services/            # سرویس‌های داخلی دامنه
│   ├── event-handlers/      # مدیریت رویدادهای داخلی
│   ├── mappers/             # تبدیل DTO به Entity و برعکس (در لایه اپلیکیشن)
│   └── scheduled-jobs/      # جاب‌های زمان‌بندی شده (Cron Jobs)
│
├── domain/                  # لایه دامنه (Business Logic Core)
│   ├── entities/            # موجودیت‌های اصلی دیتابیس و بیزنس
│   ├── events/              # رویدادهای دامنه (Domain Events)
│   └── queue.schema/        # تعریف ساختار صف‌ها
│
├── infrastructure/          # لایه زیرساخت (Implementation Details)
│   ├── mappers/             # تبدیل Entity به مدل‌های دیتابیس یا خروجی
│   ├── persistance/         # ارتباط با دیتابیس
│   │   ├── repositories/    # پیاده‌سازی مخازن داده
│   │   └── cache-server/    # تنظیمات کش
│   └── indicator/           # شاخص‌های سلامت (Health Checks)
│
└── presentation/            # لایه ارائه (API Interface)
    ├── controllers/         # کنترلرها (End-points)
    ├── dtos/                # مدل‌های انتقال داده (ورودی/خروجی API)
    ├── pipes/               # اعتبارسنجی و تبدیل داده‌های ورودی
    └── validations/         # منطق‌های اعتبارسنجی خاص
```

## توضیحات لایه‌ها (Layer Details)

### 1. Presentation Layer (لایه ارائه)

این لایه مسئول ارتباط با دنیای بیرون (HTTP Requests) است.

- **Controllers:** درخواست‌ها را دریافت کرده و به لایه Application ارسال می‌کنند.
- **DTOs (Data Transfer Objects):** ساختار داده‌های ورودی و خروجی را تعریف می‌کنند.

### 2. Application Layer (لایه کاربرد)

این لایه شامل "موارد استفاده" (Use Cases) سیستم است. در اینجا مشخص می‌شود که سیستم چه کاری انجام می‌دهد.

- **Commands:** برای عملیات‌هایی که دیتابیس را تغییر می‌دهند (Create, Update, Delete). از الگوی CQRS استفاده می‌شود.
- **Queries:** برای عملیات‌های خواندن داده بدون تغییر وضعیت.
- **Services:** منطق‌های پیچیده‌ای که در یک Command یا Query ساده نمی‌گنجند.

### 3. Domain Layer (لایه دامنه)

قلب سیستم که مستقل از فریم‌ورک و دیتابیس است.

- **Entities:** کلاس‌هایی که داده‌ها و رفتارهای اصلی بیزنس را نگه می‌دارند.
- **Events:** اتفاقاتی که در سیستم رخ داده‌اند (مثل `ActivityCheckedIn`) و دیگر بخش‌ها باید از آن مطلع شوند.

### 4. Infrastructure Layer (لایه زیرساخت)

جزئیات فنی پیاده‌سازی در اینجا قرار می‌گیرد.

- **Persistence/Repositories:** کوئری‌های مستقیم دیتابیس (Prisma) در اینجا نوشته می‌شوند تا لایه‌های بالاتر به دیتابیس وابسته نباشند.
- **Mappers:** وظیفه تبدیل داده‌ها بین لایه‌ها (مثلاً تبدیل نتیجه دیتابیس به Entity دامنه) را بر عهده دارند.

## پوشه‌های ریشه (Root Directories)

- **prisma/** : شامل فایل `schema.prisma` برای تعریف جداول دیتابیس و فایل‌های `seed` برای پر کردن داده‌های اولیه است. همچنین پوشه `migrations` برای تاریخچه تغییرات دیتابیس در اینجا قرار دارد.
- **scripts/** : اسکریپت‌های کمکی برای عملیات‌های DevOps، تست بار (Load Test) و مانیتورینگ.
